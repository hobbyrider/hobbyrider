## ‚úÖ Where to Put New Documentation

### Root Directory
**ONLY `README.md` should be in the root directory.**

### Guides (`docs/guides/`)
- User-facing guides
- Admin guides
- How-to guides
- Workflows
- Examples: `MODERATION_GUIDE.md`, `ADMIN_QUICK_REFERENCE.md`

### Setup (`docs/setup/`)
- Setup instructions
- Configuration guides
- Installation guides
- Environment setup
- Checklists
- Examples: `DEPLOYMENT.md`, `DEVELOPMENT.md`, `AUTH_SETUP.md`

### Features (`docs/features/`)
- Feature documentation
- Implementation guides
- Feature-specific setup
- Integration guides
- Service documentation
- Examples: `POSTHOG_SETUP.md`, `SENTRY_SETUP.md`, `PRODUCT_SEEDING_QUALITY_RULES.md`

### Architecture (`docs/architecture/`)
- Technical documentation
- System design
- Architecture decisions
- Code organization
- Performance documentation
- Build documentation
- Examples: `DESIGN_SYSTEM.md`, `PERFORMANCE_OPTIMIZATIONS.md`

### Troubleshooting (`docs/troubleshooting/`)
- Troubleshooting guides
- Fix documentation
- Error resolution
- Debug guides
- Examples: `DATABASE_CONNECTION_FIX.md`, `TROUBLESHOOTING_DATABASE.md`

### Planning (`docs/planning/`)
- Roadmaps
- Project plans
- Future features
- Strategy documents
- Examples: `ROADMAP.md`

### Script-Specific (`scripts/*.md`)
- Script-specific documentation stays with scripts
- Examples: `scripts/SEED_PRODUCT_INSTRUCTIONS.md`

### GitHub-Specific (`.github/*.md`)
- GitHub-specific documentation stays in `.github/`
- Examples: `.github/copilot-instructions.md`

## üö´ What NOT to Do

‚ùå **Never put `.md` files in root** (except `README.md`)  
‚ùå **Don't create random `.md` files** - use the structure above  
‚ùå **Don't mix categories** - keep related docs together  

## ‚úÖ Best Practices

1. **Use descriptive filenames**: `kebab-case.md` (e.g., `product-url-migration.md`)
2. **Update README files**: Add new docs to relevant `README.md` files
3. **Keep it organized**: Use subdirectories for related docs
4. **Link from main README**: Update `docs/README.md` with new docs

## üìù Naming Conventions

- **Guides**: Descriptive names (e.g., `admin-quick-reference.md`)
- **Setup**: Clear purpose (e.g., `deployment.md`, `auth-setup.md`)
- **Features**: Feature name (e.g., `posthog-setup.md`)
- **Troubleshooting**: Issue name (e.g., `database-connection-fix.md`)

## üîç Finding Documentation

- **Main index**: See `docs/README.md`
- **Category indexes**: Each subdirectory has its own `README.md`
- **Root README**: Links to key documentation

---

**Remember**: A well-organized documentation structure makes the project easier to navigate and maintain.
